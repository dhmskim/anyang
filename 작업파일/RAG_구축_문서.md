# RAG (Retrieval-Augmented Generation) 구축 문서

## 1. 개요

### 1.1 RAG란?
RAG(검색 증강 생성)는 AI가 답변하기 전에 관련 문서를 먼저 검색하여, 검색된 문서를 참고하여 답변을 생성하는 방식입니다.
기존 방식은 시스템 프롬프트에 모든 지식을 넣어야 했지만, RAG를 사용하면 문서 파일만 추가/수정하면 AI가 자동으로 반영합니다.

### 1.2 도입 목적
| 기존 방식 (시스템 프롬프트) | RAG 방식 |
|---------------------------|---------|
| 모든 지식을 프롬프트에 직접 작성 | 지식은 문서 파일로 분리 관리 |
| 프롬프트가 길어질수록 AI 성능 저하 | 질문과 관련된 문서만 선별하여 전달 |
| 정책 변경 시 프롬프트 직접 수정 필요 | .md 파일만 수정하면 자동 반영 |
| 지식 확장에 한계 | 문서 추가만으로 무한 확장 가능 |

### 1.3 적용 범위
- 수목원 운영안내, 주차요금, 노쇼 정책, 취소/변경, 대안주차장, 감면대상, 예약방법 등
- FAQ 성격의 모든 지식 기반 질의응답

## 2. 시스템 아키텍처

### 2.1 전체 흐름
```
사용자 질문
    ↓
[1] 질문 임베딩 (nomic-embed-text 모델)
    ↓
[2] 벡터 유사도 검색 (코사인 유사도)
    ↓
[3] 관련 문서 Top 3 추출
    ↓
[4] 시스템 프롬프트 + [참고 문서] + [잔여 현황] 조합
    ↓
[5] AI (qwen2.5:14b)가 문서 기반 답변 생성
    ↓
[6] 스트리밍 응답 → 사용자
```

### 2.2 사용 기술
| 구성 요소 | 기술 | 역할 |
|-----------|------|------|
| 임베딩 모델 | `nomic-embed-text:latest` (Ollama) | 텍스트 → 벡터 변환 |
| 생성 모델 | `qwen2.5:14b` (Ollama) | 답변 생성 |
| 벡터 검색 | 코사인 유사도 (자체 구현) | 관련 문서 검색 |
| 벡터 캐시 | `vectors.json` (파일) | 임베딩 결과 캐시 |
| Ollama 서버 | `http://172.31.0.210:11434` | AI 모델 호스팅 |

## 4. 핵심 모듈 상세 (`rag.js`)

### 4.1 문서 청킹 (splitChunks)
`knowledge/` 폴더의 .md 파일을 `##` (h2 헤딩) 기준으로 분할합니다.

```
예시: 노쇼정책.md
├── "노쇼(No-Show) 정책 > 노쇼란?" → 청크 1
├── "노쇼(No-Show) 정책 > 취소 기한" → 청크 2
├── "노쇼(No-Show) 정책 > 노쇼 제재 단계" → 청크 3
├── "노쇼(No-Show) 정책 > 노쇼 횟수 초기화" → 청크 4
├── "노쇼(No-Show) 정책 > 예외 사항" → 청크 5
└── "노쇼(No-Show) 정책 > 노쇼 방지를 위한 안내" → 청크 6
```

- 각 청크는 `문서제목 > 섹션제목 + 본문` 형식으로 구성
- 섹션이 없는 문서는 전체를 하나의 청크로 처리

### 4.2 임베딩 (getEmbedding)
Ollama의 `/api/embeddings` 엔드포인트를 사용합니다.

```
POST http://172.31.0.210:11434/api/embeddings
{
  "model": "nomic-embed-text:latest",
  "prompt": "청크 텍스트"
}
→ { "embedding": [0.012, -0.034, ...] }  // 768차원 벡터
```

- 모델: `nomic-embed-text:latest` (768차원 임베딩)
- 각 청크를 벡터로 변환하여 `vectors.json`에 캐시
- 문서 내용이 변경되지 않으면 캐시를 재사용 (빠른 재시작)

### 4.3 벡터 캐시 (`vectors.json`)
서버 시작 시 자동 생성/갱신됩니다.

```json
{
  "노쇼정책.md::노쇼란?": {
    "text": "노쇼(No-Show) 정책 > 노쇼란?\n예약 후 취소하지 않고...",
    "embedding": [0.012, -0.034, ...]
  },
  ...
}
```

- 캐시 키: `파일명::섹션제목`
- 텍스트가 동일하면 임베딩을 다시 계산하지 않음
- 문서를 수정하면 해당 청크만 재임베딩

### 4.4 유사도 검색 (search)
사용자 질문을 임베딩한 후, 모든 청크와 코사인 유사도를 계산합니다.

```
코사인 유사도 = (A · B) / (|A| × |B|)
```

- 유사도 0.3 이상인 청크만 필터링
- 상위 3개(Top-K=3) 반환
- 반환 형식:
  ```json
  {
    "source": "노쇼정책.md",
    "title": "노쇼(No-Show) 정책",
    "heading": "노쇼 제재 단계",
    "text": "...",
    "score": 0.82
  }
  ```

## 5. 서버 연동 (`server.js`)

### 5.1 서버 시작 시
```
서버 시작
  ↓
buildVectors() 실행
  ↓
knowledge/*.md 읽기 → 청크 분할 → 임베딩 → vectors.json 저장
  ↓
콘솔 출력: "[RAG] 완료: 7개 문서, 28개 청크 (캐시: 25, 신규: 3)"
```

### 5.2 채팅 요청 처리
```javascript
// 1. 사용자 메시지로 관련 문서 검색
const results = await search(message, 3);

// 2. 검색 결과를 [참고 문서]로 포맷
ragContext = '\n\n[참고 문서]\n' + results.map(r =>
    `--- ${r.title} > ${r.heading} (관련도: ${r.score}) ---\n${r.text}`
).join('\n\n');

// 3. AI에 전달하는 시스템 메시지 구성
messages = [
    { role: 'system', content: systemPrompt + ragContext + availContext },
    ...대화히스토리
];
```

### 5.3 AI에 전달되는 최종 컨텍스트 예시
```
[시스템 프롬프트 - 역할/규칙]
당신은 "서울대 안양수목원 주차장 사전예약 챗봇"입니다...

[참고 문서]
--- 노쇼(No-Show) 정책 > 노쇼 제재 단계 (관련도: 0.82) ---
| 횟수 | 제재 내용 |
| 1회 | 경고 (알림 발송) |
| 2회 | 30일간 예약 제한 |
| 3회 이상 | 90일간 예약 제한 |

--- 노쇼(No-Show) 정책 > 취소 기한 (관련도: 0.75) ---
예약일 전일(D-1) 오후 6시(18:00)까지 무료 취소 가능...

[현재 잔여 현황]
- 2026-04-03: 잔여 16면
- 2026-04-04: 잔여 9면
...
```

## 6. 지식 문서 관리 가이드

### 6.1 문서 작성 규칙
- 파일 위치: `knowledge/` 폴더
- 파일 형식: Markdown (.md)
- 제목: `# 문서 제목` (h1)
- 섹션: `## 섹션 제목` (h2) — 이 기준으로 청크가 분할됨
- 한 섹션에 너무 많은 내용을 넣지 않기 (200~500자 권장)

### 6.2 문서 추가 방법
1. `knowledge/` 폴더에 새 .md 파일 생성
2. 서버 재시작 (`start.bat`)
3. 콘솔에서 `[RAG] 완료` 메시지 확인
4. 새 문서가 자동으로 임베딩되어 검색 대상에 포함

### 6.3 문서 수정 방법
1. `knowledge/` 폴더의 .md 파일 수정
2. 서버 재시작
3. 변경된 청크만 재임베딩 (캐시된 것은 재사용)

### 6.4 현재 등록된 문서 목록
| 파일명 | 내용 |
|--------|------|
| `운영안내.md` | 운영시간, 휴원일, 위치, 입장료, 문의처 |
| `요금안내.md` | 기본요금, 감면대상별 요금, 결제방법, 환불규정 |
| `노쇼정책.md` | 노쇼 정의, 취소기한, 제재단계, 초기화, 예외사항 |
| `취소변경.md` | 취소방법, 날짜변경, 차량번호변경, 확인번호분실 |
| `대안주차장.md` | 안양예술공원, 삼성산입구, 만안구청, 대중교통 |
| `감면대상.md` | 감면대상/감면율, 등록방법, 증빙서류, 유의사항 |
| `예약방법.md` | 예약절차 6단계, 예약규칙, 주의사항 |

## 7. 시스템 프롬프트 역할 분담

### 변경 전 (RAG 도입 전)
```
시스템 프롬프트 = 역할 + 규칙 + 모든 지식 (요금, 노쇼, 운영시간, 감면 등)
→ 프롬프트가 매우 길어지고, 정보 추가 시 직접 수정 필요
```

### 변경 후 (RAG 도입 후)
```
시스템 프롬프트 = 역할 + 규칙만 (간소화)
[참고 문서]    = RAG가 질문에 맞는 문서를 자동 검색하여 주입
[잔여 현황]    = 실시간 주차장 데이터 주입
```

## 8. 성능 및 운영 참고사항

### 8.1 임베딩 소요 시간
- 최초 빌드: 약 30초~1분 (7개 문서, 약 30개 청크)
- 재시작 (캐시 있을 때): 1~2초
- 문서 1개 추가 시: 해당 문서 청크만 임베딩 (약 5초)

### 8.2 검색 소요 시간
- 질문 1건 검색: 약 0.5~1초 (임베딩 + 유사도 계산)

### 8.3 벡터 캐시 관리
- `vectors.json` 파일은 자동 관리됨
- 문서 내용이 변경되면 자동으로 해당 청크만 갱신
- 파일 삭제 시 전체 재빌드 (최초 빌드와 동일)

### 8.4 향후 확장
- 문서가 수백 개로 늘어나면 `vectors.json` 대신 벡터 DB(ChromaDB, Pinecone 등) 도입 검토
- 현재 구조는 문서 50개 이하에서 충분한 성능 발휘
- DB(MySQL/MariaDB) 연동 시 예약 트랜잭션 데이터와 RAG 지식 검색을 병행 운영
